⚙Initial one-time setup

Step 1 -- Create the Google Sheet

Create a new Google Sheet named "KLE-CTIE Internship Platform". Copy the Sheet ID from the URL (the long string between /d/ and /edit).

Step 2 -- Paste Sheet ID into Code.gs

Open Code_production.gs in Apps Script. Find line 1:

var SHEET_ID = 'PASTE_YOUR_INTERNSHIP_SHEET_ID_HERE';

Replace with your actual Sheet ID.

Step 3 -- Run setup functions (in this order)

setupSheets() // Creates 5 sheet tabs + CONFIG defaults setupTriggers() // Installs 3 daily SLA triggers seedAll() // Loads sample data for dry run (from SeedData.gs)

Step 4 -- Upload logos and set Script Properties

Upload KLE and CTIE logos to Google Drive. Make them public (Anyone with link > Viewer). Then run in Apps Script editor:

var props = PropertiesService.getScriptProperties(); props.setProperty('kle_logo_id', 'YOUR_KLE_LOGO_DRIVE_FILE_ID'); props.setProperty('ctie_logo_id', 'YOUR_CTIE_LOGO_DRIVE_FILE_ID'); props.setProperty('LOGO_KLE_URL', 'https://drive.google.com/uc?id=KLE_FILE_ID'); props.setProperty('LOGO_CTIE_URL', 'https://drive.google.com/uc?id=CTIE_FILE_ID'); props.setProperty('RESUME_FOLDER_ID', 'YOUR_RESUMES_FOLDER_ID');

Step 5 -- Deploy as Web App

Apps Script > Deploy > New Deployment > Web App.
Execute as: Me  •  Access: Anyone
Copy the /exec URL -- this is your public portal URL.

Step 6 -- Optional: Link to TBI main system

If you want the internship confirmation count to auto-sync to the TBI Impact dashboard:

props.setProperty('TBI_SHEET_ID', 'YOUR_TBI_SHEET_ID'); // Also ensure TBI CONFIG sheet has a row: intern_confirmed_count | 0
📊Day-to-day monitoring
Your daily summary The Admin Dashboard shows: total applications by status, SLA violations (red), startups with reschedule strikes, and a live internship count. Check it once a week -- the system handles everything else automatically.

What the system does automatically (you do not need to intervene)

✓ Nudges startups on days 7, 5, 24h, 48h as appropriate
✓ Auto-closes stale applications on days 14, 10, 96h
✓ Emails students when their application is closed
✓ Alerts you at 96h when a startup has not entered outcome
✓ Blocks third reschedule attempts and alerts you
✓ Syncs internship count to TBI system on every confirmation

When you need to intervene

⚠ A startup's PIN needs resetting (they emailed you)
⚠ A confirmed student is not joining (status needs manual reset)
⚠ A startup has serial SLA violations (conversation needed)
⚠ A student reports a genuine problem beyond automated handling

🏢Managing startups

Adding a new startup

Add a row to the STARTUPS sheet with all required columns. Set Active to TRUE. Leave PIN_Hash and PIN_Salt blank -- the startup sets their own PIN on first login. Share their Startup ID (format: CTIE-YYYY-NNN) with them.

Deactivating a startup

Set the Active column to FALSE in the STARTUPS sheet. Their login will be blocked and their open roles will no longer appear to students. Existing applications are preserved.

Resetting a startup PIN

Clear the PIN_Hash and PIN_Salt columns for that startup row in the STARTUPS sheet. On next login, the startup will set a new PIN automatically.

Reschedule Strikes column

Each time a startup exhausts the 2-reschedule limit, their Reschedule_Strikes counter increments. This is your indicator for which startups need a conversation about candidate experience.

📄Import and export

Importing startups in bulk

Use the Startup Master CSV template. Fill in all columns following the headers exactly (no column renames). Upload via Admin Dashboard > Import > Startups. The system validates each row before inserting.

Importing historical internship data

For historical records from before the platform, use the Applications CSV template. Set Status column to "Completed" for past internships so they count in reports without triggering SLA triggers.

Exporting for DST / NIDHI / TIDE reports

Admin Dashboard > Reports > Scheme-wise report. Filter by scheme (NIDHI-SSS, SISFS, TIDE). Download as CSV. The columns match the quarterly submission format for each scheme.

Clearing seed / test data

Run clearSeedData() from the Apps Script editor. This removes all rows from STARTUPS, INTERNSHIPS, APPLICATIONS, and PROGRESS while preserving headers. Do this before going live with real data.

🔧Configuration values
All SLA thresholds are configurable in the CONFIG sheet without touching code. Change a value in the CONFIG sheet -- it takes effect from the next trigger run.
CONFIG keyDefaultWhat it controls
max_reschedules2Maximum reschedules per application before auto-close
review_sla_days7Days before startup is nudged to review applications
review_close_days14Days before unreviewed applications auto-close
schedule_sla_days5Days before shortlisted-but-unscheduled nudge
outcome_nudge_h24Hours after interview before first outcome nudge
outcome_urgent_h48Hours before second nudge + student status change
outcome_close_h96Hours before auto-close + CTIE admin alert
otp_minutes10OTP validity duration in minutes
Note: Changing CONFIG values in the sheet does not automatically update the JavaScript constants in Code.gs. The constants in Code.gs are used as defaults when CONFIG is empty. For changes to take effect, either update both, or ensure Code.gs reads from CONFIG at runtime (future enhancement).
