01 — Why this exists

Internship coordination was running on goodwill, not process.

Before this portal, every incubated startup coordinated internships its own way — over email, WhatsApp, and loose spreadsheets. That works until it doesn't.

No visibility

CTIE had no central record of which startups were hiring, how many students had applied, or where any single application stood.

No accountability

A student could be shortlisted and then simply never hear back — no deadline existed for a startup to respond, schedule, or decide.

No audit trail

Completions, ratings, and offer confirmations lived in inboxes, not records — unusable for accreditation or incubation reporting.

02 — System design

One system of record, three roles, no new infrastructure.

The portal runs entirely inside the Institute's own Google Workspace — a Google Sheet as the system of record, Google Apps Script as the application engine. No external vendor, no data leaving KLE's own environment.

Record

The Sheet is the database

Startups, Roles, Applications, and Weekly Progress each live in their own tab — inspectable and exportable at any time, by anyone the Institute grants access to.

Engine

Apps Script runs the process

Every rule — who can see what, when a reminder fires, when an application auto‑closes — is enforced in one server‑side script, not left to manual follow‑up.

Access

Session‑based sign‑in, not shared spreadsheet access

Startups and CTIE staff never touch the raw Sheet. Each signs into a scoped portal; every privileged action is verified server‑side against their session, not assumed from the browser.

Public

Student Portal

Browse open roles with no account. Apply with an institute email. Track every stage from one link.

Startup

Startup Portal

Post roles, review applicants, schedule interviews, log weekly progress — one dashboard per startup.

CTIE

Admin Dashboard

Every application, every startup, every SLA breach — visible in real time, exportable to CSV.

05 — The differentiator

No application disappears into silence.

Every stage of the process carries a deadline. If a startup doesn't act, the system nudges them automatically — and eventually closes the application so the student is free to apply elsewhere.

Day 7

Review reminder

Startup is nudged if an application has sat unreviewed for a week.

Day 14

Auto‑close

Unreviewed applications close automatically. Student is freed to apply elsewhere.

Day 5 → 10

Scheduling window

Shortlisted candidates must be scheduled within 5 days, or the application auto‑closes at 10.

24h → 96h

Outcome window

Post‑interview, a decision is due within 48 hours — and enforced by 96.

Reschedules are capped at two per application, and offer acceptance is confirmed by a one‑time code sent to the student's institute email — not a click that can be spoofed.

09 — Security & data ownership

Not a spreadsheet with a login page bolted on.

Every privileged action — a startup viewing its applicants, CTIE pulling a report — is checked against a real server‑side session, not a value the browser is trusted to send honestly.

Session‑based authentication

Login issues a token validated on every subsequent action — a startup cannot act as another startup by guessing an ID.

PINs are hashed, never stored plain

Startup and admin PINs are salted and hashed at rest, same standard as a production credential store.

OTP‑gated offer acceptance

A one‑time code sent to the student's institute email is required to confirm an offer — no silent auto‑acceptance.

Data never leaves Workspace

No third‑party server is involved — the record of every application lives inside the Institute's own Google account.

Admin onboarding, not open sign‑up

Startups are added by CTIE directly, so every account is implicitly vetted before it exists.

Resume storage is organised, not open

Uploaded resumes are auto‑filed by year and renamed to their application ID for traceability.
